Motivation

Beauty e-commerce loses money at one specific moment

Problem 1 — Conversion

A shopper can't tell if a lipstick or foundation shade suits their skin from a flat product photo. Doubt kills the purchase.

Problem 2 — Returns

When they buy anyway and it's wrong, it comes back — eroding margin and trust.

Same root cause: you can't experience the product before buying. AR filters are gimmicks that don't map to real SKUs; shade quizzes don't show you anything. Gleame's bet: personalize with a couple of questions + a selfie, recommend real in-catalog products, and show them on the shopper's own face.

The request lifecycle

What happens in those ~20 seconds

  • ① Boot — widget fetches chat-config + recommendation-config
  • ② Consult — answers accumulate criteria = {undertone:"warm"}
  • ③ Photo — selfie uploaded → POST /chat-recommend
  • ④ Gauntlet — verify shop · billing · rate-limit · 5 MB cap
  • ⑤ Classify — Gemini reads the selfie → {depth:"fair"}
  • ⑥ Match — JSONB-equality lookup in the rules matrix, ordered by rank
  • ⑦ Render — top N try-ons in parallel (Promise.all)
  • ⑧ Backfill — a failed generation is replaced so cards are never short
  • ⑨ Resolve — batched Admin GraphQL → real product handles
  • ⑩ Cards — render → chat_recommendation_shown
A security gauntlet, a vision classification, a deterministic lookup, N parallel image generations with failure backfill, and a handle round-trip — presented to the shopper as a calm 3-step "consultation."
The recommendation engine · the insight

You can't learn from zero users

"So I don't learn preferences — I elicit them with a couple of questions and infer the rest from the photo, then map to merchant-authored rules."

Normal recommender

Needs click/purchase history → nothing to learn from on day one.

Gleame — knowledge engineering

Works at N = 0 users. Deterministic, debuggable, merchant-controlled — and upgradeable to learning later.

The recommendation engine · the match

Selfie → axes → deterministic lookup

Photo classifier — a labeler, not a generator

  • gemini-2.5-flash used as vision classifier
  • Output constrained to an enum per axis
  • temperature 0 — deterministic; can't return an unknown value
  • 768px · 12s timeout · "never refuse" · errors → {}

The match — strict JSONB equality

SELECT variant_id, product_id, rank
FROM recommendation_rules
WHERE shop_id = $1
  AND criteria = $2::jsonb
ORDER BY rank;

A rule matches or it doesn't — no fuzzy scoring. One indexed hit on UNIQUE(shop_id,criteria,rank). No match → shuffle + diversity fallback.

Multi-model, parallel, fallback-hardened

IDs abbreviated; full strings end in -preview.

Build prompt + fetch 0–5 product refs (SSRF-safe) ↓ Compress · Sharp resize · HEIC→JPEG · EXIF-rotate ↓ Gemini · 2 retries w/ backoff ↓ on failure (with refs) OpenAI · safety-framed prompt
Parallel top-N generation (Promise.all) → wall-clock ≈ one image. Backfill → the shopper always gets a full set. No output caching — every result is a function of this photo (hit rate ≈ 0).
The hard problem

The latency · cost · quality triangle

  • Latency → parallel transforms, input downscaling, capped retries, a consultation UI that makes the wait feel purposeful
  • Cost → flash by default, pro only where precision pays; generate only what you show; rate limits as a circuit-breaker
  • Quality → real product reference images, deep-skin visibility strategy, pro model for makeup precision
Even optimized: an estimated ~20s wait, mostly on mobile (perceived; end-to-end latency not yet instrumented), before value appears. that's the leak
Unit economics & margin

Flat revenue, variable cost

Revenue — flat

Session-based subscription tiers (Hey Mantle):

Free $0 · Starter $30 · Launch $149 · Growth $399

Merchant pays the same whether shoppers run 10 try-ons or 10,000.

Cost — variable, per session

cost = N×image-gen + 1×classification

N = 3 by default (configurable 1–5). Scales with every shopper who completes the flow.

Per-image $ lives on the provider's rate card — never guessed in the room.

This is the tension that drives the architecture: model tiering, "only render what you show," and rate limits are margin decisions — not just latency ones.
Prompt generation

Deterministic, not LLM-written

categories.csv→ category_parameters.csv→ parameter_levels.csv+ variant_color_profiles→ final transform instruction

How

Merchant picks a level per parameter → system stitches each level's prompt_text + locked "guardrail" prompts + the variant's color profile (shade, hue family, undertone, deep-skin strategy) into one string.

Why it's a feature

Reproducible (same inputs → same prompt), merchant-controlled (they own brand voice + safety), and no prompt-injection surface (no free-text reaches the model uncontrolled).

The synthesis

"On desktop, where the photo step is frictionless, the try-on lifts conversion and basket size — the concept directionally validates. On mobile, photo friction costs marginal converters faster than the try-on wins them back — and mobile is most of the traffic. The AI works. The mobile friction is the bug. That's a product fix, not a model fix."
